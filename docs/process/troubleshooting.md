# Troubleshooting

Common issues and their resolutions, drawn from accumulated operational
experience across 57+ releases.

## SSH Key Failures

### Provider SSH: "attempted methods [none]"

**Cause:** The bpg/proxmox provider's Go SSH library only parses RSA keys. It
SSHes to PVE to upload cloud-init snippets (no API endpoint for this).

**Fix:** Ensure `~/.ssh/id_rsa` exists (RSA, not ed25519) and the RSA public
key is in `secrets.yaml` under `ssh_keys`. For delegated PVE nodes,
`CopySSHPrivateKeyAction` handles this automatically.

### PVE Replaces Root's authorized_keys

**Cause:** PVE manages `/root/.ssh/authorized_keys` and may overwrite additions.

**Fix:** Use `vm_user` (default: `homestak`) for provider SSH instead of root
(iac-driver#133). The `homestak` user's `authorized_keys` is not managed by PVE.

## Stale Server

**Symptom:** VMs bootstrap with old code or specs despite recent commits.

**Cause:** The iac-driver server daemon serves code from the installed repos.
After `homestak update`, the server still runs with old code in memory.

**Fix:** Restart the server:

```bash
cd ~/iac/iac-driver
./run.sh server stop
./run.sh server start
```

## Provider Version Mismatch

**Cause:** Stale provider binaries or lockfiles in `.state/tofu/*/data/providers/`
after Dependabot bumps or manual version changes.

**Fix:** Clear cached providers and re-run:

```bash
rm -rf ~/.state/tofu/*/data/providers/
```

### State Version 4 Error

**Symptom:** `OpenTofu does not support state version 4, please update.`

**Cause:** OpenTofu bug (opentofu/opentofu#3643) when `terraform.tfstate` exists
inside `TF_DATA_DIR`. iac-driver works around this by storing state at the
parent level. For manual tofu usage, keep state outside `TF_DATA_DIR`.

## Apt Lock Contention

**Cause:** Another apt process is running (cloud-init `package_update` or
`unattended-upgrades` at boot).

**Fix:** Bootstrap and ansible handle this with retry logic. For manual runs,
wait for the lock holder to finish (`sudo lsof /var/lib/dpkg/lock-frontend`).

## Site Configuration Issues

### Gateway or DNS Not Configured

**Symptom:** Preflight fails with "gateway not configured" or "dns_servers not
configured".

**Cause:** `site.yaml` still has empty defaults from the template.

**Fix:** Edit `~/config/site.yaml`:

```yaml
defaults:
  gateway: "192.0.2.1"        # your network gateway
  dns_servers: ["192.0.2.1"]   # your DNS servers
```

### Missing Node Config

**Cause:** No `nodes/<hostname>.yaml` exists (generated by `homestak pve-setup`).

**Fix:** Run `homestak pve-setup` on the target host.

## Missing Packer Images

**Cause:** Cloud images not downloaded/published to PVE storage.

**Fix:** `homestak images download all --publish` (downloads from GitHub releases
to `local:iso/`).

## SSH Host Key Changed

**Cause:** Target was reprovisioned, generating a new SSH host key.

**Fix:** `ssh-keygen -R <hostname>` then reconnect.

## Integration Test Failures

### Prerequisites Checklist

Before running integration tests (`manifest test`), verify:

1. Code is committed and merged to master
2. Target host is updated: `homestak update`
3. `secrets.yaml` has: API token, signing key, RSA pubkey in `ssh_keys`
4. Packer images are published
5. `site.yaml` has gateway and dns_servers configured
6. Server is running with current code: `./run.sh server status`

### Running Tests

```bash
ssh jderose@srv1 'sudo -u homestak bash -c "
  cd ~/iac/iac-driver && ./run.sh manifest test -M n1-push -H srv1 --verbose
"'
```

Use `--verbose` to get detailed output for diagnosing failures.
